**会议纪要关键信息** (同一建物会议内容.txt):
1. 执行时间: 毎月3日早朝6時
2. 对象: 前月1日〜末日に入居実績がある全利用者
3. 数据来源:
   - 共同生活援助 → Kintoneの外泊・入院実績シート（入居日/退去日/月別集計）
   - 住宅型有料/特定施設/認知症GH/特養 → kintone居出変更管理アプリ(k/197/)
4. 同一建物ルール: 同じ建物に2つのサービスがある場合、合併して1施設として登録
5. 月途中転居: A施設→B施設の場合、2レコード出力
6. 同名利用者: 複数行出る可能性あり（異なるサービス事業所）→ careuid で区別必要
7. 利用者マッチングキー: 名前完全一致
8. スプレッドシートは毎月シートを増やす形式
9. ヘッダ: 入居施設、あおぞらID、利用者名、入居日、退去日、新規フラグ、ステータス、備考
10. 実際のヘッダには「利用訪問看護事業所名」も含まれる（4列目）
11. 最悪シートへの出力だけでも初回は手打ちで対応可能

**RPA操作フロー**（HTML分析済み）:
- Step 1: TRITRUSポータル → 同一建物管理画面 (`/tritrus/premisesIndex/index`)
- Step 2: 施設一覧から対象施設の`transferPremisesUpdate(id)`ボタンクリック
- Step 3: 施設詳細 → `openCareuserWindow()` で利用者追加弾窗
- Step 4: 弾窗内テーブルで事業所名+利用者名マッチ → checkbox
- Step 5: `addCareuserToMain()` で追加確定
- Step 6: スプレッドシートにステータス書き戻し

**重要注意**: これはHAMのiframe内ではなく、TRITRUSポータル直下のページ

## Kintone参考代码分析

**现有GAS脚本** (Google Apps Script) — 每日执行，目标表格: `1hQSmIO5q1EhNzuKhdbjiaz0EGwQu290JNOekAS5bJkY`

### 数据源结构:
1. **Kintone App 197 (居室利用変更履歴管理)**
   - 字段: あおぞらID, 入居希望者名, 契約適用開始日, 退去日, 入居施設名, 居室番号, 提供事業
   - 当前只筛选 `入居施設名 = "梅ヶ丘・特養"` → 同一建物需要扩展到全施設
   - API Token存在Google Drive文件中: `1CKtiEYb49LnGmQH0oRYlitBOOS23cvZi`

2. **Kintone App 184 (入退院管理)**
   - 字段: あおぞらID, 入院日, 退院日, 対象施設名
   - 用于更新入院状态（E/F列）
   - 当前也只筛选梅ヶ丘・特養

3. **Kintone App 182 (特養ショートステイ管理)**
   - 字段: あおぞらID, ご利用者様名, 開始日, 終了日
   - Token直接硬编码: `WGgkEs2VlKRtW5LqNyqI9e8CU9CkZ5KK5Ov0gLK3`
   - 写入 `利用者一覧(短期)` sheet

### GAS脚本的main()流程:
1. getAndSyncRecordsToSheet() — 从kintone 197获取入居者 → 写入利用者一覧sheet
2. updateHospitalizationData() — 从kintone 184获取入退院数据 → 更新E/F列
3. syncShortStayUsersToNewSheet() — 从kintone 182获取短期入住 → 写入利用者一覧(短期)
4. cleanNamesInUserList() — 清理名前（去除末尾「様」、删除空格）

### 关键发现:
- GAS脚本用GAS的UrlFetchApp调用Kintone REST API
- 当前只处理梅ヶ丘・特養，同一建物管理需要全部施设
- 退去日过去的记录会被skip
- 退院日是未来日期的视为尚未退院（空白处理）
- 名前清理: 去「様」+ 删除全角/半角空格

### 数据流:
```
Kintone 197 (全施设) ──┐
Kintone 184 (入退院)  ──┤→ 利用者一覧sheet → 同一建物管理連携sheet(18DueDsY...)
Kintone 182 (短期)   ──┘
共同生活援助GHスプレッドシート ──→ (月別集計/入居日/退去日から)
```
