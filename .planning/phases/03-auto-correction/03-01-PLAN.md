**Method 2: `runAutoCorrection()`** — Orchestrates the full correction cycle (D-01).

```typescript
/**
 * 自動修正: 不一致レコードの削除→再転記 (FIX-01, FIX-02)
 * 1サイクルのみ実行 (D-08)。例外発生時はそのレコードをスキップして続行 (D-09)。
 */
private async runAutoCorrection(
  location: SheetLocation,
  mismatches: VerificationMismatch[],
  records: TranscriptionRecord[],
  nav: HamNavigator,
  dryRun: boolean,
  tab?: string,
): Promise<{ corrected: string[]; failed: string[]; nav: HamNavigator }>
```

Implementation:
1. Build a recordId -> TranscriptionRecord map from `records` for quick lookup.
2. Separate mismatches into two groups per D-03/D-04:
   - `toDelete`: mismatches where `!mm.missingFromHam` (time/service/staff — need HAM deletion)
   - `toResetOnly`: mismatches where `mm.missingFromHam === true` (just reset T-column)
3. Track arrays: `corrected: string[]`, `failed: string[]`, `allResetIds: string[]`
4. For each `toDelete` mismatch (D-03 loop):
   - Find the TranscriptionRecord by recordId from the map
   - If record has no `hamAssignId`, log warning "assignId不明 — 自動修正スキップ" and add to `failed`, continue
   - Wrap in try-catch (D-09):
     - Call `this.deleteHamRecord(nav, record)` — if returns false, add to failed, continue
     - Call `this.sheets.resetForRetranscription(location.sheetId, record.rowIndex, tab)`
     - Add recordId to `allResetIds`
     - Log: `logger.info(\`[自動修正] 削除完了: ${record.patientName} ${record.visitDate}\`)`
   - Catch: log error, add to `failed`, recover via `nav = await this.tryRecoverToMainMenu(nav)`
5. For each `toResetOnly` mismatch (D-04):
   - Find record, wrap in try-catch
   - Call `this.sheets.resetForRetranscription(location.sheetId, record.rowIndex, tab)`
   - Add recordId to `allResetIds`
   - Log: `logger.info(\`[自動修正] ステータスリセット(missingInHam): ${record.patientName} ${record.visitDate}\`)`
   - Catch: add to failed
6. If `allResetIds.length > 0` and not dryRun:
   - Re-transcribe by calling `this.processLocation(location, dryRun, tab, allResetIds)`
   - This reuses the existing transcription loop with targetRecordIds filtering
   - Add IDs from the returned result (processedRecords > 0) to `corrected`
   - Any errors from result go to `failed`
7. Return `{ corrected, failed, nav }`

IMPORTANT: The `processLocation` recursive call will itself call `runVerification` at the end, which handles FIX-03 (re-verification). However, the re-transcription's runVerification will again detect mismatches and could trigger another runAutoCorrection. To prevent infinite loops per D-08, add a class-level field:
```typescript
private _correctionCycleActive = false;
```
At the start of `runAutoCorrection`, set `this._correctionCycleActive = true`. The calling code (in Plan 02) will check this flag before calling runAutoCorrection.

Actually, simpler approach: the method that CALLS runAutoCorrection (in Plan 02) will only call it once. The re-transcription via processLocation will run verification but NOT auto-correction (because Plan 02 will gate the auto-correction call behind a flag). So the recursion guard is in Plan 02, not here.

For now, just implement the method without recursion guards — Plan 02 will handle the orchestration.

NOTE on re-transcription approach: Instead of calling `this.processLocation()` recursively (which would re-fetch records, re-login, etc.), a simpler approach is to just let the corrected records be picked up in the NEXT run of the workflow. But per D-01, we want same-session correction. So the recursive processLocation call with targetRecordIds is the right approach. The nested processLocation will:
- Re-fetch records from Sheets (with reset T-column = 未転記)
- Filter to targetRecordIds only
- Run the transcription loop
- Run verification at the end

The nav (HamNavigator) needs to be valid for the nested call. Since processLocation calls `this.auth.ensureLoggedIn()` at the start, the session will be reused.
